## 概要

AI エージェントに「記憶」を持たせる方法には、表面的に機能しているように見えて実際には劣化し続ける「偽のメモリ」と、本当にスケールする「真のメモリ」がある。両者の違いを理解することで、長期稼働エージェントの品質崩壊を防げる。

## 偽のメモリ（Markdown ダンプ）

セッションの会話や気づきを Markdown ファイルに書き出して Claude Code に読み込ませる方式。一見「記憶している」ように動くが、以下の問題がある:

| 問題 | 内容 |
|------|------|
| **重複蓄積** | 同じ情報が形を変えて何度も記録される |
| **検索不能** | キーワードしか検索できず、意味的な関連を辿れない |
| **鮮度管理なし** | 古い情報と新しい情報が同じ重みで読まれる |
| **関係モデリングなし** | エンティティ間のつながりが表現できない |
| **コンテキスト肥大化** | ファイルが増えるほど読み込みトークンが増え、やがて Context Rot を引き起こす |

Markdown メモはあくまで「人間が読むメモ」であり、AI がランタイムで効率的に検索・更新するデータ構造としては不適切。

## 真のメモリが必要とする 4 要素

| 要素 | 内容 | 実装技術 |
|------|------|---------|
| **重複排除** | 同一・類似情報を統合して一意性を保つ | セマンティックハッシュ・埋め込み比較 |
| **セマンティック検索** | 意味的な類似度でクエリに応答 | ベクトル DB（Chroma, Pinecone, pgvector）|
| **時間加重** | 最近の記憶を優先し、古い記憶を減衰 | タイムスタンプ × 重みスコア |
| **関係モデリング** | エンティティ間のつながりを表現 | ナレッジグラフ（Neo4j, GraphRAG）|

## 実装ソリューション

### Mem0

エージェント向けのメモリ管理ライブラリ。会話から自動的にエンティティ・関係・事実を抽出し、重複排除・時間加重した上で保存する。OpenAI / Anthropic の API と統合可能。

### GraphRAG

Microsoft が開発した RAG の拡張。従来の「チャンク→ベクトル」ではなく、ドキュメントからエンティティと関係を抽出してナレッジグラフを構築する。複数文書にまたがる複雑な推論に強い。

### memsearch

LLM エージェント向けの意味検索インフラ。Markdown ファイル群に対してセマンティック検索レイヤーを追加するシンプルな構成。Markdown ベースのシステムから移行しやすい。

## 設計の判断基準

| ケース | 推奨アーキテクチャ |
|--------|------------------|
| プロトタイプ・短期実験 | Markdown ダンプでも許容（スケールしない前提） |
| 長期稼働・パーソナライズ | Mem0 または同等のセマンティックメモリ |
| 複数文書の複雑な関係推論 | GraphRAG |
| 既存 Markdown 資産の活用 | memsearch で検索レイヤーを追加 |
