## 概要

Claude Code などの AI コーディングエージェントを使うと、短期的に開発速度が上がる一方で、見えにくいコストが積み上がる。「AI に全部任せると速い」という認識が誤りである理由を整理する。

## 主な隠れたコスト

### 1. 技術的負債の高速蓄積

AI は「動くコード」を素早く生成するが、「保守しやすいコード」かは別問題。

- 命名・構造・パターンが統一されていないコードが増える
- テストなしで動く実装が積み上がる
- 将来の変更コストが指数的に増加する

「速く作れた」分だけ、後始末のコストが増える。

### 2. AI 生成コードのレビューコスト

AI が出力したコードを人間がレビューするのは、ゼロから書くより難しいケースがある:

- コードの意図が分かりにくい（なぜこの実装を選んだか）
- バグが微妙（動くが端末ケースで壊れる）
- 量が多い（AI は過剰に実装しがち）

「AI にレビューさせればよい」という考えも、AI 同士のレビューがハルシネーションを見逃す問題がある。

### 3. コンテキスト管理コスト

長いセッションでは [Context Rot](/blogs/wiki/concepts/context-rot/) が発生し、出力品質が低下する。

- セッション管理・リセット・要約の運用コスト
- CLAUDE.md・Skills のメンテナンスコスト
- プロジェクト固有のルールを常に最新に保つ負担

### 4. API コスト

Claude API の課金は意外と積み上がる:

- 大型コードベースの読み込みトークン
- 複数エージェントの並列実行
- 失敗→リトライのトークン無駄遣い

Plan Mode の活用でトークン 64%・コスト 69% の削減が可能（詳細: [Claude Code](/blogs/wiki/tools/claude-code/)）。

### 5. チーム教育・プロセス変更コスト

AI コーディングを導入する際に見落とされやすい人的コスト:

- チームメンバーの習熟時間
- CLAUDE.md・ハーネス設計の共有・統一
- 「AI が生成したコードをどうレビューするか」のガイドライン策定
- 既存の開発フロー（PR レビュー・テスト・デプロイ）との整合

## バランスの取り方

| 戦略 | 内容 |
|------|------|
| **ハーネス設計を先行** | CLAUDE.md・Hooks・検証ゲートを整備してから AI を全力活用 |
| **Plan Mode の徹底** | 実装前の設計フェーズでコストと品質を確定させる |
| **テストを先に書く** | AI に実装させる前に期待する振る舞いをテストで表現する |
| **段階的な自動化** | いきなり全部 AI に任せず、影響範囲の小さい部分から試す |
